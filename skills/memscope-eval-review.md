# MemScope 评测体系审阅 Skill

## 一、赛题需求 Checklist

### 1.1 四大方向覆盖度

- [ ] **方向 A: CLI 高频命令与工作流记忆** — 显式+隐式记忆，自动记录高频命令模式并推荐
- [ ] **方向 B: 飞书项目决策与上下文记忆** — 结构化记忆+时序关联+主动推送，从对话/文档中自动提取决策
- [ ] **方向 C: 个人工作习惯与偏好记忆** — 隐式学习+规则生成+主动服务，通过观察用户行为自动学习偏好
- [ ] **方向 D: 团队知识断层与遗忘预警** — 团队共享+遗忘曲线+版本管理，遗忘时主动"复习提醒"

### 1.2 三项交付物

- [ ] **《Memory 定义与架构白皮书》** — 明确记忆场景、数据流向图、价值论证
- [ ] **可运行的 Demo** — 基于 OpenClaw 和飞书 API 的实际代码
- [ ] **自证评测报告 (Benchmark Report)** — 包含抗干扰测试、矛盾更新测试、效能指标验证

### 1.3 评测报告三项必测项

- [ ] **抗干扰测试** — 大量无关对话后，仍能精准捞取一周前注入的关键记忆
- [ ] **矛盾更新测试** — 先后输入冲突指令，证明系统理解时序并正确覆写记忆
- [ ] **效能指标验证** — 量化成果（如字符节省率 80%）

### 1.4 五项评分要点

- [ ] 记忆场景定义是否清晰
- [ ] 系统架构设计（提取/存储/检索/更新/遗忘）是否完整
- [ ] 评测数据质量（覆盖抗干扰、矛盾更新、效能验证）
- [ ] 实际效能证明（真实数据）
- [ ] 飞书生态集成（与 OpenClaw/飞书 API 的集成）

## 二、评测理论基准

### 2.1 LongMemEval (ICLR 2025)

- 500 个评测实例，7 种能力分类
- 信息提取 IE（single-session-user/assistant/preference）
- 多会话推理 MR、知识更新 KU、时间推理 TR、拒答 ABS
- 主要指标：QA 准确率（LLM-as-judge, GPT-4o）
- 次要指标：Recall@k 和 NDCG@k
- 关键发现：商用系统准确率下降 37%-64%

### 2.2 LOCOMO (ACL 2024)

- 10 个长对话，1,813 个问题，最多 35 个会话，平均 300 轮/对话
- 4 种问题类别：Single-hop, Multi-hop, Temporal, Adversarial
- 关键发现：对抗性问题对模型特别困难

### 2.3 MemScope 评测方案 v2.0 定义的核心指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 命中率 Hit Rate | >= 85% | 搜索结果中包含目标信息的比例 |
| 精确率 Precision | >= 85% | 搜索结果中相关信息的比例 |
| 召回率 Recall | >= 90% | 目标信息被检索到的比例 |
| F1-Score | >= 87% | 精确率和召回率的调和平均 |
| 操作节省率 | >= 50% | 记忆推荐减少的操作步骤比例 |
| 字符节省率 | >= 60% | 记忆推荐减少的输入字符比例 |
| 时间节省率 | >= 40% | 记忆推荐节省的时间比例 |
| 写入延迟 P50 | <= 200ms | 记忆写入操作的中位延迟 |
| 查询延迟 P50 | <= 300ms | 记忆查询操作的中位延迟 |
| 并发 | >= 10 ops/sec | 系统并发处理能力 |

## 三、评测代码审阅 Checklist

### 3.1 存储层真实性

- [ ] 评测是否使用真实的 SqliteStore 而非简化存储（MiniStore/mock）
- [ ] 搜索是否使用真实的混合搜索引擎（FTS5 + 向量 + Pattern + RRF + MMR）
- [ ] 每个测试用例是否使用独立的存储实例（避免交叉污染）
- [ ] 数据库 schema 是否与生产环境一致

### 3.2 评测通过条件逻辑

- [ ] 每个评估函数是否返回 `failed_checks` 字段
- [ ] pass/fail 判定是否基于实际指标值（而非空列表检查）
- [ ] recall=0.0 / chunks_found=0 的情况是否被正确判定为 fail
- [ ] 是否存在"永远 pass"的评估函数（返回字典无 failed_checks）

### 3.3 指标计算正确性

- [ ] recall 计算是否基于搜索结果与期望关键词的匹配
- [ ] precision 计算是否排除了噪声关键词
- [ ] F1 是否正确计算为 2*P*R/(P+R)
- [ ] 效率指标是否测量了真实的写入/查询延迟
- [ ] 操作节省率/字符节省率/时间节省率是否有具体计算公式

### 3.4 评测覆盖度

- [ ] 是否覆盖了全部 8 个评测维度
- [ ] 每个维度是否有独立的评估函数
- [ ] 权重分配是否与 evaluation_scheme_v2.md 一致
- [ ] 是否包含长时序（3月-2年）多轮对话样本
- [ ] 是否包含难度分层（Easy 30% / Medium 50% / Hard 20%）

### 3.5 报告输出机制

- [ ] 每次运行评测是否自动产出 JSON 结果文件
- [ ] 是否有从 JSON 自动生成 Markdown 报告的代码
- [ ] 报告是否包含：执行摘要、各维度得分、详细测试结果、基准对标、改进建议
- [ ] 报告中的数字是否全部来自实际运行（非硬编码）

### 3.6 Hermes Agent 连接

- [ ] MemScopeProvider 是否实现了 Hermes Agent 的 memory_provider 接口
- [ ] 是否通过 ctx.register_memory_provider() 注册
- [ ] plugin.yaml 是否正确声明了 hooks 和配置
- [ ] 评测是否在真实 Hermes Agent 运行时环境中执行
- [ ] 是否验证了 prefetch() / sync_turn() / on_session_end() 生命周期

### 3.7 飞书集成

- [ ] 是否有真实的飞书 API 调用代码（非模拟/mock）
- [ ] 是否支持飞书互动卡片推送（决策卡片、知识健康预警）
- [ ] 是否支持中英文双语决策提取
- [ ] CLI 终端和飞书端是否可以自由切换
- [ ] 是否有 FEISHU_APP_ID / FEISHU_APP_SECRET 的真实认证流程

## 四、数据集质量审阅 Checklist

### 4.1 格式一致性

- [ ] 所有数据集文件是否使用统一的 JSON 结构
- [ ] 每条样本是否包含：test_id, name, difficulty, setup, query, expected
- [ ] expected 字段结构是否统一（各维度是否有差异）
- [ ] total_cases 声明数量是否与实际 test_cases 数组长度一致

### 4.2 样本独立性

- [ ] 各样本之间是否存在事实关联（应该彼此独立）
- [ ] 是否使用了统一的虚拟人名/项目名（避免跨样本依赖）
- [ ] 时间戳是否合理（不同时序场景的样本使用不同时间段）

### 4.3 难度分层

- [ ] Easy/Medium/Hard 分布是否接近 30%/50%/20%
- [ ] Easy 样本是否测试基本功能
- [ ] Medium 样本是否测试多步骤/多条件场景
- [ ] Hard 样本是否测试边界情况/高噪声/长时序

### 4.4 长时序覆盖

- [ ] 是否包含短时序样本（1天-1周）
- [ ] 是否包含中时序样本（1周-3个月）
- [ ] 是否包含长时序样本（3个月-2年）
- [ ] 长时序样本是否有足够的干扰项（"大海捞针"场景）

### 4.5 主流 Benchmark 对标

- [ ] 是否参考了 LongMemEval 的 7 种能力分类
- [ ] 是否参考了 LOCOMO 的 4 种问题类别
- [ ] 抗干扰测试是否包含：单轮/多轮/高相似度/时间跨度/角色混淆干扰
- [ ] 矛盾更新测试是否包含：直接覆盖/部分更新/时间线矛盾/多实体并发/撤回取消

## 五、常见评测陷阱与反模式

### 5.1 "永远 Pass" 陷阱

评估函数返回的字典中没有 `failed_checks` 字段，主循环用 `metrics.get("failed_checks", [])` 判定，空列表视为 pass。导致 recall=0.0 也标记为 pass。

### 5.2 "代码测试 = 性能评估" 反模式

用 pytest 的思路来做性能评估：只检查函数是否抛异常、返回值是否非空，而不检查返回值的语义正确性。

### 5.3 "模拟 = 真实" 反模式

评测脚本使用 mock/stub/simplified 实现，而非真实的生产代码路径。评测结果无法代表真实系统性能。

### 5.4 "精选数据" 反模式

只展示通过的测试结果，隐藏失败的测试；或只用精心设计的小规模场景（15 条对话）而非全量数据集（240 条）。

### 5.5 "指标缺失" 反模式

只统计通过率（pass rate），不计算覆盖率、准确率、召回率、F1 等具体指标；或只报告延迟 P50 而不报告 P95/P99。
