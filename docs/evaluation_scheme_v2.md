# MemScope 评测方案 v2.0

> 基于 LONGMEMEVAL/LOCOMO 公开基准理论支撑 + 赛题实际要求
> 更新时间：2026-04-29

---

## 1. 总体框架

### 1.1 评测目标

依据赛题「挑战三：证明它的价值 (Prove it)」要求，通过自证评测报告证明 MemScope 系统：
1. **真的"记住了"** — 命中率、召回率
2. **产生了实际效能** — 操作节省率、字符节省率、步数对比

### 1.2 评测维度与权重

| 维度 | 权重 | 对应赛题方向 | 评测数据集 |
|------|------|-------------|-----------|
| 抗干扰能力 | 15% | 赛题必测项 | anti_interference (30条) |
| 矛盾信息更新 | 15% | 赛题必测项 | contradiction_update (30条) |
| 效率指标 | 15% | 赛题必测项 | efficiency (30条) |
| CLI命令记忆 | 10% | 方向A | command_memory (30条) |
| 飞书决策记忆 | 15% | 方向B | decision_memory (30条) |
| 个人偏好记忆 | 15% | 方向C | preference_memory (30条) |
| 团队知识健康 | 10% | 方向D | knowledge_health (30条) |
| 长时序记忆 | 5% | 跨方向综合 | long_term_memory (30条) |

**总计**: 8 个数据集，240 条评测样本

### 1.3 评分标准

- **满分**: 100 分
- **及格线**: 70 分
- **优秀线**: 85 分

## 2. 评测数据集制作规范

### 2.1 样本独立性原则

- 每个样本**彼此独立**，无事实和逻辑上的关联
- 不同样本之间不共享公司名、人名、项目名等实体
- 避免样本间形成"记忆泄露"（一个样本的输入恰好是另一个样本的答案）

### 2.2 业务场景贴合

每个样本必须结合**具体公司业务场景**：
- 使用真实的公司部门名称（技术部、产品部、市场部等）
- 使用真实的业务术语（API接口、数据库迁移、CI/CD流水线等）
- 使用真实的项目场景（版本发布、客户对接、技术选型等）

### 2.3 时间跨度要求

- **短时序**: 1天~1周（抗干扰、矛盾更新）
- **中时序**: 1周~3个月（命令记忆、决策记忆）
- **长时序**: 3个月~2年（长期记忆、知识健康）

### 2.4 难度分层

每个数据集包含三级难度：
- **Easy (30%)**: 单轮干扰、直接覆盖、简单查询
- **Medium (50%)**: 多轮干扰、部分更新、跨会话推理
- **Hard (20%)**: 高相似度干扰、时间推理、多实体并发

### 2.5 样本结构规范

```json
{
  "test_id": "dataset_name_NNN",
  "name": "样本名称",
  "difficulty": "easy|medium|hard",
  "category": "子类别",
  "description": "样本描述（一句话说明测试什么）",
  "setup": {
    "context": "业务背景描述",
    "conversations": [
      {
        "role": "user|assistant",
        "content": "对话内容",
        "timestamp": "ISO 8601"
      }
    ]
  },
  "query": {
    "text": "查询文本",
    "type": "search|frequent|recommend|health_check|..."
  },
  "expected": {
    "answer": "期望答案",
    "keywords": ["必须包含的关键词"],
    "forbidden": ["禁止出现的关键词"],

      "hit_rate": 1.0,
      "precision_min": 0.85,
      "recall_min": 0.90
    }
  }
}
```

## 3. 评测指标体系

### 3.1 核心指标（赛题要求）

#### 3.1.1 命中率 (Hit Rate)

```
命中率 = 命中样本数 / 总样本数 × 100%
```

- **定义**: 系统返回结果中包含正确答案（或正确答案的关键词）的样本比例
- **判定规则**: 期望关键词在返回结果内容中出现即为命中
- **目标**: ≥ 85%

#### 3.1.2 精确率 (Precision)

```
Precision = 返回结果中正确信息数 / 返回总信息数
```

- **定义**: 系统返回结果中，正确信息占总返回信息的比例
- **目标**: ≥ 85%

#### 3.1.3 召回率 (Recall)

```
Recall = 正确召回的目标字段数 / 总目标字段数
```

- **定义**: 系统能够回忆起的目标信息比例
- **目标**: ≥ 90%

#### 3.1.4 F1-Score

```
F1 = 2 × Precision × Recall / (Precision + Recall)
```

- **目标**: ≥ 87%

### 3.2 效能指标（赛题要求）

#### 3.2.1 操作节省率

```
操作节省率 = (使用前操作步数 - 使用后操作步数) / 使用前操作步数 × 100%
```

- **测量**: 对比有记忆系统和无记忆系统完成同一任务的操作步数
- **目标**: ≥ 50%

#### 3.2.2 字符节省率

```
字符节省率 = (使用前输入字符数 - 使用后输入字符数) / 使用前输入字符数 × 100%
```

- **测量**: 对比有记忆系统推荐和无记忆系统时用户需要输入的字符数
- **目标**: ≥ 60%

#### 3.2.3 时间节省率

```
时间节省率 = (使用前完成时间 - 使用后完成时间) / 使用前完成时间 × 100%
```

- **目标**: ≥ 40%

### 3.3 系统性能指标

#### 3.3.1 写入延迟

- P50 ≤ 200ms, P95 ≤ 500ms, P99 ≤ 1000ms

#### 3.3.2 查询延迟

- P50 ≤ 300ms, P95 ≤ 800ms

#### 3.3.3 并发吞吐

- ≥ 10 ops/sec

## 4. 各数据集评测细则

### 4.1 抗干扰测试 (anti_interference, 30条)

**赛题要求**: "在输入大量无关对话/操作后，系统依然能精准捞取一周前注入的关键记忆"

**评测场景**:
- 单轮噪声干扰 (6条: 3 easy, 2 medium, 1 hard)
- 多轮连续噪声干扰 (6条: 2 easy, 3 medium, 1 hard)
- 高相似度干扰 (6条: 2 easy, 2 medium, 2 hard)
- 时间跨度干扰 (6条: 2 easy, 2 medium, 2 hard)
- 角色混淆干扰 (6条: 2 easy, 2 medium, 2 hard)

**评测指标**: 命中率、精确率、召回率、F1、噪声引入率

### 4.2 矛盾信息更新 (contradiction_update, 30条)

**赛题要求**: "先后输入两条冲突的指令，证明系统能理解时序，正确覆写记忆"

**评测场景**:
- 直接覆盖型矛盾 (6条)
- 部分更新型矛盾 (6条)
- 时间线矛盾 (6条)
- 多实体并发矛盾 (6条)
- 撤回/取消型矛盾 (6条)

**评测指标**: 最新值准确率、历史保留率、时间排序准确率

### 4.3 效率指标 (efficiency, 30条)

**赛题要求**: "量化展示成果（使用前需要敲50个字符，使用后只需10个，提效80%）"

**评测场景**:
- 写入延迟测试 (6条: 不同长度对话)
- 查询延迟测试 (6条: 不同数据量级)
- 内存占用测试 (6条)
- Token消耗效率 (6条)
- 并发性能测试 (6条)

**评测指标**: P50/P95/P99延迟、内存占用、Token效率、吞吐量

### 4.4 CLI命令记忆 (command_memory, 30条)

**对应方向A**: "自动记录用户的高频命令模式、常用参数组合、项目路径偏好"

**评测场景**:
- 高频命令识别 (6条: 单用户/多用户交叉)
- 命令推荐 (6条: 前缀补全/上下文感知)
- 项目路径关联 (6条: 不同项目不同命令集)
- 上下文感知推荐 (6条: 根据当前项目推荐)
- 操作节省验证 (6条: 对比有/无记忆的输入字符数)

**评测指标**: 命中率、字符节省率、推荐准确率

### 4.5 飞书决策记忆 (decision_memory, 30条)

**对应方向B**: "从对话/文档中自动提取决策、理由、反对意见、结论"

**评测场景**:
- 中文决策提取 (8条: "定下来"、"确认"、"采用"等信号词)
- 英文决策提取 (4条: "decided"、"confirmed"等)
- 决策搜索 (6条: 按关键词/项目搜索历史决策)
- 决策卡片推送 (6条: 相关话题触发历史决策)
- 决策生命周期 (6条: 记录→推翻→更新)

**评测指标**: 命中率、提取准确率、推送相关性

### 4.6 个人偏好记忆 (preference_memory, 30条)

**对应方向C**: "通过观察用户行为，自动学习其偏好和规律"

**评测场景**:
- 显式偏好提取 (8条: "我喜欢X"、"我习惯用Y")
- 隐式偏好推断 (6条: 从行为模式推断)
- 偏好更新 (6条: 偏好变化后的正确覆写)
- 上下文感知推荐 (6条: 根据时间/场景推荐)
- 偏好冲突解决 (4条: 显式vs隐式冲突)

**评测指标**: 命中率、偏好准确率、更新准确率

### 4.7 团队知识健康 (knowledge_health, 30条)

**对应方向D**: "允许团队注入需要长期记住的事项，设置遗忘曲线"

**评测场景**:
- 知识新鲜度检测 (6条: fresh/aging/stale/forgotten)
- 遗忘预警 (6条: 基于Ebbinghaus曲线)
- 知识缺口检测 (6条: 10大领域覆盖分析)
- 团队知识同步 (6条: 信息更新后团队同步)
- 关键知识遗忘 (6条: 高重要性知识的遗忘检测)

**评测指标**: 新鲜度准确率、预警及时性、缺口检测率

### 4.8 长时序记忆 (long_term_memory, 30条)

**综合评测**: 跨越2年的长时间记忆能力

**评测场景**:
- 3个月~6个月回忆 (8条)
- 6个月~1年回忆 (8条)
- 1年~2年回忆 (8条)
- 跨年度信息更新 (6条: 信息在不同年份发生变化)

**评测指标**: 命中率、时间推理准确率

## 5. 评测流程

### 5.1 评测执行流程

```
[加载评测数据集] → [初始化MemScope系统] → [逐样本执行评测]
                                                    ↓
[写入setup数据] → [执行query] → [收集返回结果] → [计算指标]
                                                    ↓
[汇总各维度得分] → [生成评测报告] → [输出JSON + Markdown]
```

### 5.2 评测脚本要求

1. **全量评测**: 必须运行全部 240 条样本，不得选取部分
2. **独立执行**: 每个样本使用独立的存储实例，避免数据污染
3. **规则评测**: 使用代码和规则判定结果，不依赖 LLM 判断
4. **结果持久化**: 评测结果输出为 JSON 文件，包含每条样本的详细指标

### 5.3 评测报告格式

```json
{
  "evaluation_id": "eval-YYYYMMDD-NNN",
  "timestamp": "ISO 8601",
  "system_version": "MemScope vX.Y.Z",
  "total_cases": 240,
  "passed": N,
  "failed": N,
  "pass_rate": "XX.X%",
  "overall_score": XX.X,
  "dimension_scores": {
    "anti_interference": {"score": XX.X, "weight": 0.15, "details": {...}},
    ...
  },
  "efficiency_metrics": {
    "write_latency_p50_ms": X.XX,
    "query_latency_p50_ms": X.XX,
    "operation_saving_rate": "XX%",
    "character_saving_rate": "XX%"
  },
  "cases": [...]
}
```
